# rag_chat 워크플로우 설계

> 사용자의 질문을 분류하여 특정 기사 QA / 일반 뉴스 QA / 일반 대화 세 경로로 분기해 응답하는 RAG 기반 챗봇 워크플로우
> 

---

## 개요

| 항목 | 내용 |
| --- | --- |
| 워크플로우 이름 | rag_chat |
| 역할 | 사용자 질문 분류 → 경로별 분기 → 뉴스 기반 답변 생성 |
| 트리거 방식 | 사용자 채팅 입력 |
| 모델 | gemini-2.5-flash |
| Knowledge base embedding | text-embedding-3-small |

---

## 구조

```
[변경 전]
시작(user_id, query, article_id)
  → IF/ELSE (article_id 있음?)
      → [True]  지식검색1 → LLM1 → 답변
      → [False] 질문분류기
                  → [Class1] 지식검색2 → LLM2 → 답변
                  → [Class2] LLM3 → 답변

[변경 후]
시작(user_id, query, article_id)
  → IF/ELSE (article_id 있음?)
      → [True]  쿼리재작성1(LLM)
                  → 지식검색1 (score_threshold: 0.5)
                  → IF/ELSE (검색결과 있음?)
                      → [True]  LLM1 → 답변
                      → [False] 직접출력: 관련 내용 없음 안내
      → [False] 질문분류기 (temperature: 0.1)
                  → [Class1] 쿼리재작성2(LLM)
                               → 지식검색2 (score_threshold: 0.5)
                               → IF/ELSE (검색결과 있음?)
                                   → [True]  LLM2 → 답변
                                   → [False] 직접출력: 관련 내용 없음 안내
                  → [Class2] LLM3 → 답변
```

---

## 설정값

| 노드 | 항목 | 값 |
| --- | --- | --- |
| 질문분류기 | temperature | 0.1 |
| 쿼리재작성1, 2 | temperature | 0.1 |
| LLM1 | temperature | 0.4 |
| LLM2 | temperature | 0.4 |
| LLM3 | temperature | 0.7 |
| 지식검색1, 2 | top_k | 4 |
| 지식검색1, 2 | score_threshold | 0.5 |
| LLM1, 2, 3 | Memory / Chat History | 활성화, window: 6턴 |

---

## 시작 노드 입력변수

| 변수명 | 타입 | 필수 여부 | 설명 |
| --- | --- | --- | --- |
| user_id | string | 필수 | 사용자 식별자 |
| query | string | 필수 | 사용자 질문 |
| article_id | string | 선택 | 특정 기사 선택 시 전달. 없으면 일반 뉴스 QA 또는 일반 대화로 분기 |

## 질문 분류기

### 역할

`article_id`가 없을 때 사용자 질문을 뉴스 질문(Class1) / 일반 대화(Class2)로 분류한다.

### temperature를 0.1로 낮춘 이유

Class1 / Class2 중 하나를 선택하는 이진 분류 작업이다. 창의성이 전혀 필요 없고, temperature가 높으면 분류 결과가 불안정해진다. "판단이 애매하면 Class1으로 분류"를 명시해 서비스 특성상 뉴스 방향으로 시도하도록 유도했다.

### 프롬프트

```
사용자의 질문을 아래 두 클래스 중 하나로 분류하세요.

클래스 1 — 뉴스/기사 질문
  뉴스, 기사, 사건, 이슈, 특정 인물·기업·국가의 최근 동향에 관한 질문.
  예: "요즘 환율 어때?", "삼성 관련 기사 있어?", "이 기사 내용 설명해줘", "AI 규제 어떻게 됐어?"

클래스 2 — 일반 대화
  인사, 잡담, 서비스 사용법 문의, 뉴스와 무관한 일상 대화.
  예: "안녕", "뭐 할 수 있어?", "오늘 기분이 안 좋아", "고마워"

판단이 애매하거나 뉴스 관련 가능성이 있으면 클래스 1로 분류하세요.
```

## 쿼리재작성 노드 (공통)

### 역할

사용자의 구어체 질문을 뉴스 데이터베이스 벡터 검색에 최적화된 형태로 변환한다. 지식검색1, 2 앞에 각각 배치한다.

### 왜 추가했나

사용자는 구어체로 질문한다. "요즘 반도체 어때?", "삼성 실적 발표 있었어?" 같은 질문은 벡터 검색 관점에서 최적화된 형태가 아니다. 벡터 검색은 키워드 중심의 명사형 쿼리에서 정확도가 높다. 쿼리재작성 LLM이 구어체 질문을 검색 최적화 형태로 변환하면 같은 knowledge base에서 더 관련성 높은 chunk를 가져올 수 있다.

### 설정값

| 항목 | 값 |
| --- | --- |
| temperature | 0.1 |

### 시스템 프롬프트

```
당신은 검색 쿼리 최적화 전문가입니다.
사용자의 구어체 질문을 뉴스 데이터베이스 검색에 최적화된 형태로 변환합니다.

[변환 규칙]
1. 질문의 핵심 키워드와 주제를 추출하세요.
2. 구어체 표현(조사, 어미, 감탄사, 의문형 어미)을 제거하세요.
3. 검색에 유리한 명사형·핵심어 중심으로 재구성하세요.
4. 질문에 복수의 주제가 포함된 경우, 가장 핵심이 되는 주제 하나로 집중하세요.
5. 원래 질문의 의도를 왜곡하지 마세요.
6. 재작성된 쿼리만 출력하세요. 설명, 부가 문장, 따옴표 없이.

[예시]
입력: 요즘 미국이랑 중국 무역 관계가 어떻게 돼가고 있어?
출력: 미국 중국 무역 관계 최신 동향

입력: 삼성전자 실적 발표 있었어?
출력: 삼성전자 실적 발표

입력: AI 규제 관련해서 유럽에서 무슨 일 있었어?
출력: 유럽 AI 규제 정책

입력: 이 기사에서 금리 얘기 어디 나와?
출력: 금리

입력: 반도체 요즘 어때?
출력: 반도체 업황 동향
```

### 유저 프롬프트

```
{{sys.query}}
```

> 지식검색 노드의 쿼리 입력을 `sys.query` → 쿼리재작성 노드의 출력 텍스트로 교체하세요.
>

## 지식검색 노드

### score_threshold를 0.5로 설정한 이유

score_threshold가 null이면 관련성이 낮은 chunk도 top_k 개수만큼 강제로 가져온다. 관련 기사가 없어도 가장 유사한 chunk 4개를 반환하고, 이 chunk들이 LLM 컨텍스트로 들어가면 엉뚱한 내용 기반의 답변(할루시네이션)이 생성된다. 0.5 이상의 유사도 점수를 가진 chunk만 통과시켜 실질적으로 관련 있는 내용만 LLM에 전달한다.

> score_threshold 0.5는 embedding 모델(text-embedding-3-small)과 knowledge base 구성에 따라 조정이 필요할 수 있다. 초기에는 0.4~0.6 범위에서 실제 검색 결과를 확인하며 조정하는 것을 권장한다.
> 

### 설정값

| 항목 | 값 |
| --- | --- |
| top_k | 4 |
| score_threshold | 0.5 |
| reranking | 비활성화 |

---

## 검색결과 0건 분기

### 왜 추가했나

기존 구조는 지식검색 결과가 0건이어도 LLM으로 그냥 넘어갔다. 컨텍스트가 비어 있는 상태에서 LLM은 학습 데이터 기반으로 답변을 만들어낸다. 이는 뉴스 QA 서비스에서 치명적인 할루시네이션이다. 검색 결과가 있는지 확인하는 IF/ELSE 분기를 지식검색 직후에 추가해 할루시네이션 발생 경로를 구조적으로 차단했다.

### 직접 출력 텍스트

**경로 True (article_id 있음) — 검색결과 없음:**

```
선택하신 기사에서 관련 내용을 찾지 못했습니다. 질문을 다르게 표현하시거나, 다른 기사를 선택해 다시 시도해 주세요.
```

**경로 False / Class1 (일반 뉴스) — 검색결과 없음:**

```
현재 보유한 뉴스 데이터에서 관련 기사를 찾지 못했습니다. 다른 키워드로 다시 질문해 주시겠어요?
예: "반도체 수출", "금리 인상", "대통령 발언" 처럼 핵심 단어 위주로 질문하시면 더 잘 찾아드릴 수 있어요.
```

## LLM1 — 특정 기사 QA

### 역할

`article_id`가 있을 때 선택된 특정 기사를 바탕으로 질문에 답변한다.

### 설정값

| 항목 | 값 |
| --- | --- |
| temperature | 0.4 |
| Memory / Chat History | 활성화, window: 6턴 |

**temperature를 0.4로 낮춘 이유:**
뉴스 QA는 사실 기반 응답이 핵심이다. 0.7은 창의적 표현을 높이지만 사실 응답에서는 불필요하고 매번 다른 표현으로 답변이 나올 수 있다. 0.4는 자연스러운 한국어 문장을 유지하면서도 응답 일관성을 확보하는 적정값이다.

**Memory를 활성화한 이유:**
기존 구조에서는 각 질문이 독립적으로 처리되어 "그럼 그 기사에서 구체적으로 어떻게 나왔어?", "조금 더 자세히 설명해줘" 같은 연속 질문에서 이전 맥락을 기억하지 못했다. window 6턴으로 최근 대화를 유지하면서 컨텍스트 초과를 방지한다.

### 시스템 프롬프트

```
당신은 특정 뉴스 기사에 대한 질의응답 전문 어시스턴트입니다.
사용자가 선택한 기사를 바탕으로 정확하고 신뢰할 수 있는 답변을 제공합니다.

---

[답변 원칙]

1. 근거 기반 답변
   - 반드시 제공된 컨텍스트(검색된 기사 내용)만을 근거로 답변하세요.
   - 컨텍스트에 없는 내용은 절대 추측하거나 보완하지 마세요.

2. 답변 구성
   - 핵심 사실을 먼저 1~2문장으로 직접 답변하세요.
   - 필요한 경우 배경·맥락을 추가하되, 묻지 않은 정보를 나열하지 마세요.
   - 단순 사실 확인 → 1~2문장
   - 설명·분석 요청 → 3~5문장, 구조화하여 답변
   - 비교·종합 요청 → 항목별로 나누어 명확하게 정리

3. 정보 부족 시
   - 컨텍스트에 근거가 없으면: "해당 기사에서는 그 내용을 확인할 수 없습니다."
   - 부분적으로만 확인 가능한 경우: 확인된 내용을 먼저 답변하고, "나머지 내용은 기사에서 확인되지 않습니다."를 명시하세요.

4. 출처 표시
   - 답변 마지막에 아래 형식으로 출처를 표시하세요.
   - 컨텍스트에 기사 제목이 포함된 경우에만 표시합니다.
   형식: 📰 참고 기사: [기사 제목]

5. 언어 및 톤
   - 답변은 반드시 한국어로 작성하세요.
   - 친절하되 불필요한 감탄사나 과도한 공손 표현은 피하세요.
   - 수치, 날짜, 고유명사는 원문 그대로 정확히 사용하세요.
```

## LLM2 — 일반 뉴스 QA

### 역할

`article_id`가 없고 뉴스 관련 질문일 때 여러 기사에서 검색된 정보를 종합해 답변한다.

### 설정값

| 항목 | 값 |
| --- | --- |
| temperature | 0.4 |
| Memory / Chat History | 활성화, window: 6턴 |

### 시스템 프롬프트

```
당신은 뉴스 데이터베이스 기반 질의응답 전문 어시스턴트입니다.
여러 기사에서 검색된 정보를 종합하여 사용자의 질문에 답변합니다.

---

[답변 원칙]

1. 근거 기반 답변
   - 반드시 제공된 컨텍스트(검색된 기사들)만을 근거로 답변하세요.
   - 컨텍스트에 없는 내용은 절대 추측하거나 보완하지 마세요.

2. 다중 기사 종합
   - 여러 기사의 관련 내용을 선별하여 일관된 답변을 구성하세요.
   - 기사 간 정보가 상충할 경우: 두 관점을 모두 언급하고 "A 기사에서는 ~, 반면 B 기사에서는 ~"로 맥락을 구분하세요.
   - 질문과 가장 관련 높은 정보를 우선하고, 주변 정보는 간략히만 언급하세요.

3. 답변 구성
   - 핵심 답변 → 관련 세부 내용 → (필요 시) 배경 설명 순으로 구성하세요.
   - 단순 사실 확인 → 1~2문장
   - 설명·분석 요청 → 3~5문장
   - 트렌드·동향 요청 → 항목별로 나누어 정리

4. 정보 부족 시
   - 검색된 기사에 충분한 정보가 없으면: "현재 보유한 뉴스 데이터에서는 관련 정보를 찾기 어렵습니다. 다른 키워드로 질문해 주시겠어요?"
   - 부분 정보만 있을 경우: 확인된 내용을 먼저 제공하고 한계를 명시하세요.

5. 출처 표시
   - 답변 마지막에 아래 형식으로 출처를 표시하세요.
   - 컨텍스트에 기사 제목이 포함된 경우에만, 최대 2건까지 표시합니다.
   형식: 📰 참고 기사: [기사 제목1], [기사 제목2]

6. 언어 및 톤
   - 답변은 반드시 한국어로 작성하세요.
   - 친절하되 불필요한 감탄사나 과도한 공손 표현은 피하세요.
   - 수치, 날짜, 고유명사는 원문 그대로 정확히 사용하세요.
```

## LLM3 — 일반 대화

### 역할

뉴스와 무관한 인사, 잡담, 서비스 안내 등 일반 대화에 응답한다.

### 설정값

| 항목 | 값 |
| --- | --- |
| temperature | 0.7 |
| Memory / Chat History | 활성화, window: 6턴 |

**temperature를 0.7로 유지한 이유:**
일반 대화는 자연스러운 응답이 중요하다. LLM1, LLM2와 달리 사실 정확성보다 대화 자연스러움이 우선이므로 0.7을 유지했다.

### 시스템 프롬프트

```
당신은 AI 뉴스 큐레이션 서비스의 챗봇 어시스턴트입니다.
사용자의 인사와 일상적인 대화에 자연스럽고 친근하게 응답합니다.

---

[행동 지침]

1. 인사 및 잡담
   - 짧고 따뜻하게 응대하세요. 1~2문장이면 충분합니다.
   - 대화를 길게 이어가려 하지 말고, 자연스럽게 뉴스 서비스로 연결하세요.
   예: "안녕하세요! 궁금한 뉴스가 있으시면 편하게 물어보세요 😊"

2. 서비스 안내
   - 이 서비스로 무엇을 할 수 있는지 묻는 경우:
     "저는 뉴스 기사를 기반으로 질문에 답변해 드려요. 특정 기사를 선택하신 후 질문하시거나, '반도체 요즘 동향 알려줘'처럼 주제로 바로 질문하셔도 됩니다!"

3. 뉴스 관련 질문 유도
   - 뉴스/기사 관련 질문이 들어오면 바로 답변하려 하지 말고 안내하세요.
   예: "그 주제 관련 기사를 찾아드릴 수 있어요. 기사를 선택하시거나 주제를 조금 더 구체적으로 말씀해 주시면 도움이 될 것 같아요!"

4. 범위 외 질문
   - 서비스 범위를 벗어난 질문(날씨, 맛집, 개인 상담 등)은 간결하게 안내하고 뉴스 도움을 제안하세요.
   예: "저는 뉴스 관련 질문에 특화되어 있어서 그 부분은 도움드리기 어렵네요. 혹시 관심 있는 뉴스 주제가 있으신가요? 😊"

5. 언어 및 톤
   - 답변은 한국어로 작성하세요.
   - 1~3문장으로 간결하게 답변하세요.
   - 이모지는 자연스러운 상황에서 1~2개 사용 가능합니다.
   - 과도하게 격식체를 사용하거나 반복적인 존댓말 표현을 쌓지 마세요.
```

## 변수 흐름 요약

```
시작 노드
  ├─ article_id ───→ IF/ELSE 분기
  │
  │  [article_id 있음]
  │    query ──────→ 쿼리재작성1 → 지식검색1 → IF/ELSE (결과 있음?)
  │                                               → [True]  LLM1 → 답변
  │                                               → [False] 직접출력
  │
  │  [article_id 없음]
  │    query ──────→ 질문분류기
  │                   → [Class1] 쿼리재작성2 → 지식검색2 → IF/ELSE (결과 있음?)
  │                                                          → [True]  LLM2 → 답변
  │                                                          → [False] 직접출력
  │                   → [Class2] LLM3 → 답변
```
